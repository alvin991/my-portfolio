Homeschool Planner

A full-stack Next.js application for managing a homeschool curriculum: draggable lesson/folder outlines, enrollment scheduling with a live calendar preview, a month-view calendar with per-student auto-rescheduling, shared resources (subjects, publishers), and a GraphQL API over MongoDB. The codebase is structured so the UI, API boundary, and data model stay explicit — useful as a portfolio piece for discussing system design in interviews.

Tech Stack

Also used: Apollo Server + Client, Mongoose, dnd-kit, Vitest.

Features

  • Draggable lesson/folder outline (dnd-kit) with a course template system where enrollments copy lessons for per-student isolation.
  • Resync function that propagates teacher updates from the course template to enrolled students without overwriting their progress.
  • Enrollment scheduling (weekdays, week interval, suspension periods, optional end date) with a live calendar preview before saving.
  • Month and day calendar views with subject color coding and a status workflow (pending, completed, skipped) editable from a lesson popover.
  • Lazy per-student auto-rescheduling: overdue lessons are detected and pushed forward on calendar load, no background job required.
  • GraphQL API over MongoDB, with resolvers split by domain (courses, resources, calendar) and merged into a single schema.
  • Shared resource management for subjects, publishers, and students.

Challenges

  • Preview vs. saved schedule drift
    Challenge: Enrollment previews and actual saves need to produce identical schedules, or the calendar the user previews could mislead them.
    Resolved: computeSchedule is a single pure function called from both createEnrollment and previewEnrollmentSchedule; the preview path runs the exact computation and simply never writes to the database.
  • Auto-rescheduling without a cron job
    Challenge: Overdue lessons need to shift forward automatically, but a background job adds infrastructure and staleness windows.
    Resolved: processOverdueLessons runs on calendar-view queries, scanning active enrollments in memory and only writing to the database when overdue lessons are actually found — imperceptible latency for an app opened daily.
  • Keeping outline UI and saved structure in sync
    Challenge: The sidebar tree and lesson/folder forms can drift apart after drag-and-drop or optimistic UI updates.
    Resolved: A lessonTreeSourceRef alongside React state ensures saves read the same tree the outline renderer displays.
  • Live vs. preview calendar data fetching
    Challenge: The same calendar grid components are reused between the live calendar and the enrollment preview modal, but their data can change under different rules.
    Resolved: The live calendar re-queries on every month change since saved data can change underneath it; the preview modal fetches the full projected schedule once and filters client-side, since an unsaved schedule can't change out from under the user.
  • Resisting premature DRY
    Challenge: The month-view calendar resolver and the enrollment preview both compute scheduled dates, tempting a shared abstraction.
    Resolved: Kept them separate — the month view has different concerns (multiple enrollments, month filtering, reconciling live status) that would have added branching to the simpler, single-enrollment preview path just to share code.

Learnings

  • Practiced modeling a recursive tree (lesson outline) as an embedded Mongoose subdocument and weighing that against normalization — fewer round-trips and transactional consistency vs. single-document growth limits.
  • Learned to pair parallel arrays by index (scheduled_dates and lesson_occurrences) to keep lookups O(1) without a join.
  • Reinforced that sharing a pure computation (not just UI components) is what actually prevents preview/save drift, rather than sharing the surrounding resolver logic.
  • Gained comfort documenting scope honestly — noting where authentication and multi-tenant isolation were deliberately left out of a portfolio project rather than glossing over it.
  • Built experience structuring a GraphQL schema and resolvers by domain so each module owns its own Query/Mutation fields without overwriting siblings.
